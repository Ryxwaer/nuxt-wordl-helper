## §5. Sitemap warnings - real issue ⚠

`sitemaps.list()` returns:

```json
{
  "path": "https://wordl.ryxwaer.com/sitemap.xml",
  "lastSubmitted": "2025-07-09",
  "lastDownloaded": "2026-05-17",
  "warnings": "2",
  "errors": "0",
  "contents": [{"type": "web", "submitted": "3", "indexed": "0"}]
}
```

Two things stand out:

1. **`indexed: 0` of `submitted: 3`.** This is a **sitemap-level credit** stat, not the indexing reality (URL inspection confirms all 3 are indexed). Google is indexing the URLs, but is *not* attributing the indexing decision to the sitemap. That's a hygiene flag, not a crawl-blocker.
2. **2 sitemap warnings** that the API doesn't expose textually. Most likely candidates given our setup:
    - `lastmod` set to build-time stays static for weeks → Google flags as "lastmod not changing despite changefreq=daily".
    - The `urls()` callback in `nuxt.config.ts` runs once at prerender time, so `lastmod = new Date().toISOString()` resolves to the build timestamp, not request time. The sitemap currently shows `2026-04-24T12:52:09Z` for all three URLs - same as the last commit before the 1dae6a2 deploy.

**Fix path:** generate the sitemap at request-time (not prerender) so `lastmod` truly reflects "now". `@nuxtjs/sitemap` supports `runtimeCacheStorage` and `prerender: false` for this. Action item §11.

## §6. Theme-driven long-tail - considered and rejected

**Original framing (since revised):** the 28-day query list shows that every WOTD theme generates multi-impression search traffic for weeks *after* it rotates away - TriFi SuperApp alone produced 75+ impressions over 28 days despite no longer being the active theme. The naïve recommendation was to build a `/themes/[slug]` archive page per past theme to capture this traffic.

**Why this was wrong (added 2026-05-24 after user pushback):** Binance rotates the WOTD theme weekly and the previous week's word pool is *not* in play. If a player lands on `/themes/trifi-superapp`, sees a "word pool" published as if it were authoritative, and pastes those words into today's WOTD challenge, **every guess fails**. They lose their daily streak because of our content. That isn't an SEO trade-off; that's active reputational harm.

The [calculator analogy that resolved this](#) is also the right product principle for the whole site: the app is a tool, not a content directory. We don't add pages for `5+5`, `5+6`, `5+7` just because users Google those individually. We don't build keyword-harvest pages, even when the data shows the queries exist.

**Concrete record of the data we are choosing not to capture:**

| Theme (week) | Sample query | 28d impr | 28d clicks |
|---|---|---|---|
| TriFi SuperApp | `trifi superapp binance word of the day` | 5 | 2 |
| TriFi SuperApp | `trifi superapp wotd` | 17 | 1 |
| TriFi SuperApp | `binance wodl trifi superapp` | 19 | 1 |
| TriFi SuperApp | `"trifi superapp" binance word of the day` | 20 | 0 |
| TriFi SuperApp | `"trifi superapp" word of the day` | 11 | 0 |
| Binance Chat | `binance wordle "binance chat"` | 8 (7d) | 0 |
| Binance Chat | `"binance chat" word of the day solution` | 16 | 0 |
| Prediction Markets | `"word of the day" binance "prediction markets" 6 letters` | 8 | 0 |

**These 75+ stale-theme impressions are real but not actionable.** Note that of the impressions that actually *converted* (4 clicks total in the table), all happened during the active week of that theme - not after rotation - so the loss from refusing to build archives is even smaller than the impression count suggests.

The principle is now codified in `CLAUDE.md` under **Product principles**. Future audit recommendations to "build /themes archive", "add per-length pages", "add localized landing pages", or any other keyword-harvest variant should be rejected on the same grounds without re-litigating.

### What we *do* keep from this finding

The **active-week** theme integration on `/wodl` (dynamic title, current theme word pool, theme words ranked first in the solver) is working - `binance wordle "binance chat"` ranks #1 right now because the current week's theme is Binance Chat. Keep it. It serves users who are mid-game *this* week. The `/wodl` page never publishes stale pools because it always shows the live API result.

## §9. The `x-powered-by: Nuxt` mystery

```
$ curl -sI https://wordl.ryxwaer.com/
server: openresty
x-powered-by: Nuxt
x-served-by: wordl.ryxwaer.com
```

Our `server/plugins/strip-headers.ts` Nitro plugin (committed `e75ec9a`, deployed for 3+ weeks) calls `deleteResponseHeader(event, 'x-powered-by')` in the `beforeResponse` hook. Code is correct.

The header still appearing - combined with `server: openresty` (which is *not* what Nitro returns, Nitro returns `server: cloudflare` once proxied) - strongly suggests the architecture is:

```
client → Cloudflare → OpenResty (your reverse proxy) → Nuxt/Nitro
```

**OpenResty is adding `x-powered-by: Nuxt` independently of Nuxt** - likely via an `add_header X-Powered-By Nuxt;` directive in your nginx config, or via `proxy_pass_header X-Powered-By` forwarding it from a default nginx setup that injected it. Either way, **the fix is in your OpenResty config, not in Nuxt code.**

In nginx terms (whichever config block proxies to Nuxt):

```nginx
proxy_hide_header X-Powered-By;
more_clear_headers 'X-Powered-By';   # if ngx_headers_more is loaded
add_header X-Powered-By "" always;   # last-resort override
```

This is a minor info-leak; SEO impact is essentially zero. But since it's been flagged 4 weeks running and our code-side fix is correct, it's worth resolving at the proxy.
